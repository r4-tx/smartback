Diagrama de classes de domínio

Atende ao item: diagrama de classes de domínio, com as principais classes da análise do sistema e como se relacionam. O desenho abaixo está alinhado às entidades JPA (com.smartstock.entity) — ou seja, ao que de fato persiste no PostgreSQL, não só ao esboço conceitual.

O que dizer na apresentação (requisito da disciplina)

O trabalho pede o diagrama de classes de domínio e a explicação das principais classes da análise e dos relacionamentos. Use este slide após as telas: primeiro mostramos como o usuário vê, depois como o sistema modela o mesmo negócio por baixo.

Domínio é o vocabulário do negócio traduzido em classes. Aqui não estamos desenhando “telas”: estamos mostrando quais conceitos o sistema persiste e quem se relaciona com quem. Isso prova que o mini-ERP não é só interface: existe modelo de dados coerente atrás.

Observe três ideias centrais: cadastros mestres (cliente, produto, fornecedor, local), o agregado de venda (Order + OrderItem) e os efeitos colaterais auditáveis (StockMovement e Delivery).

  • UUID como chave: adequado para APIs distribuídas e merge de dados; no código está em todas as entities.
  • Valores monetários em BigDecimal no Java — evita erro de arredondamento de ponto flutuante em preço e total.
  • OrderItem mantém productId (UUID), não um @ManyToOne Product: é um desenho pragmático; o produto continua sendo entidade independente.
  • Supplier e StockLocation aparecem como cadastros à parte: no schema atual não há FK delas em Product — o menu do ERP já antecipa módulos que podem evoluir para vínculo explícito depois.
Dica de banca: se perguntarem “cadastro X conversa com Y?”, responda com o que existe no banco hoje (FK / coluna) versus o que é roteiro de evolução. Isso é maturidade de arquiteto de software, não desculpa.

Legenda técnica: Order ↔ OrderItem = @OneToMany / @ManyToOne com cascade no agregado. Supplier e StockLocation existem no banco mas não possuem FK para Product no modelo atual — por isso sem linhas.

Cartões rápidos — use como legenda ao apontar no diagrama:

Client Entity

Quem compra. CPF/CNPJ único. O pedido guarda só clientId — desacoplamento: dados do cliente podem ser consultados sem acoplar objeto grafado em todo lugar.

Product Entity

SKU comercial + stock atual. Movimentações e itens de pedido sempre apontam para o mesmo id; o saldo físico “mora” aqui.

Order Agregado

Cabeçalho da venda: número legível (G1…), status de ciclo de vida, total. Itens são filhos com cascade no JPA — apagar pedido remove linhas órfãs.

OrderItem Entity

Linha concreta: quantidade, preço na hora da venda, desconto, subtotal. Snapshot parcial do preço — histórico não quebra se o produto mudar de preço depois.

StockMovement Auditoria

Cada entrada/saída fica registrada com origin (ex.: “Pedido #G3”) e stockAfter. Isso explica “por que o estoque está X” sem reprocessar todos os pedidos.

Delivery Logística

Ponte entre venda e entrega física: cidade, data programada, status operacional. Liga orderId e clientId para montar a fila de rotas na UI.

Supplier / StockLocation

Cadastros de apoio ao ERP “completo” da navegação. Úteis na demo de telas; evolução natural é FK opcional (ex.: produto ← fornecedor padrão).

Sequência sugerida ao explicar o diagrama (2–3 min)
  1. Mostre Client e Product como base — “sem isso não existe operação”.
  2. Centralize em Order: “aqui nasce o documento de venda”; abra OrderItem como composição.
  3. Trace o arco de estoque: pedido → StockMovement + atualização de Product.stock (no serviço, transacional).
  4. Feche com Delivery: mesmo pedido, outro subdomínio — separação de conceitos entre financeiro/comercial e logística.

Arquitetura em camadas

SPA no navegador + API REST + banco relacional: o Next.js cuida de interface e navegação; o Spring Boot expõe recursos HTTP e executa regras e transações; o PostgreSQL persiste o modelo de domínio. O browser nunca abre conexão JDBC — só JSON via HTTP (lib/api.ts → localhost:8081/api).

Visão de arquiteto

Separação em camadas reduz acoplamento: amanhã você pode trocar o front (ou publicar um app mobile) consumindo a mesma API. A lógica que não pode ser burlada pelo cliente — validar pedido, baixar estoque, gerar entrega — fica no servidor, idealmente em métodos anotados com @Transactional para garantir consistência (tudo confirma ou tudo desfaz).

No backend, o fluxo típico é: Controller (mapeia verbo e URL para um caso de uso) → Service (orquestra repositórios e regras) → Repository (Spring Data JPA) → PostgreSQL. No front, páginas em app/ e componentes por módulo chamam funções tipadas que encapsulam fetch.

  • Front: apresentação, estado de tela, validação só para UX (campos obrigatórios, máscaras).
  • API: contrato estável (recursos como /clients, /orders); cabeçalhos e status HTTP semânticos.
  • Persistência: entidades JPA refletem tabelas; em dev, ddl-auto: update; em produção madura, migrações versionadas.
Base da API: variável NEXT_PUBLIC_API_URL (padrão http://localhost:8081/api). Em deploy, aponta para o host real — o código do front permanece o mesmo.

Exemplo de fala — um request ponta a ponta

“Quando o usuário salva um pedido, o React chama POST /api/orders. O OrderController delega ao OrderService, que em uma transação valida cliente e itens, persiste o pedido, grava as saídas em stock_movements, atualiza products.stock e insere a linha em deliveries. A resposta JSON repopula a lista na tela. Ou deu certo em conjunto, ou o banco desfaz — é isso que significa transação.”

Fluxo de negócio integrado

O diagrama abaixo é a história que o sistema conta: do cadastro ao painel. O ponto crítico é que criar pedido não é só inserir uma linha — dispara efeitos em estoque e logística, implementados no serviço de domínio.

Leitura para a banca

Em ERP real, “pedido” é um evento de negócio. O SmartStock modela isso de forma didática: ao confirmar a venda, o estoque precisa refletir a saída física e a entrega precisa aparecer na fila do entregador — mesmo que a UI seja simplificada.

O retângulo verde “pedido salvo” marca o commit conceitual do documento; laranja e ciano mostram efeitos colaterais obrigatórios desse mesmo passo no código (OrderService.create).

Pré-condições

Cliente e produto existentes. Pedido referencia IDs válidos — o serviço responde 404 se algo não existir.

Decisão de estoque

Hoje o serviço ajusta com Math.max(0, stock - qtd); na apresentação, mencionem se pretendem evoluir para bloquear venda sem estoque — regra de produto, não detalhe de framework.

Uma transação

Pedido + movimentações + entrega devem ser vistos como pacote atômico. Se o banco falhar no meio, nada fica “pela metade” — conceito ACID.

Pós-venda

Entrega e dashboard são leituras diferentes do mesmo dado consolidado: logística atualiza status; o painel agrega contagens e receita.

Aba 1 — Telas / protótipos (~3 min)

“Vamos percorrer as principais telas do protótipo: login, dashboard, cadastro de clientes e produtos, criação de pedido, lista de pedidos, movimentações de estoque e entregas. A ordem segue o fluxo de uso de um mini-ERP, do acesso ao encerramento logístico.”

“Cada tela consome a API REST; no vídeo vale mostrar um pedido sendo salvo e, em seguida, o mesmo impacto aparecendo em estoque e entregas.”

Aba 2 — Diagrama de classes (~2,5 min)

“Este é o modelo de domínio do SmartStock, desenhado a partir das classes que o Hibernate persiste no PostgreSQL. Em vez de começar pela tela, a gente mostra quais conceitos o sistema realmente guarda.”

“Temos cadastros mestres: cliente e produto. O pedido é o agregado central: ele tem cabeçalho e uma lista de itens. Cada item aponta para um produto e guarda quantidade e valores na hora da venda — isso é importante para histórico: se o preço do produto mudar amanhã, o pedido antigo continua correto.”

“Quando a venda impacta o físico, não basta mudar um número: a gente grava movimentação de estoque com origem e saldo depois do movimento. Assim dá para auditar. Em paralelo, nasce uma entrega ligada ao pedido e ao cliente, que a tela de rotas atualiza até ‘entregue’.”

“Fornecedor e local de estoque estão no mesmo desenho porque o menu do ERP já prevê esses cadastros; no banco atual eles são independentes do produto — deixamos explícito para a banca que não estamos inventar relacionamento que não existe no código.”

Aba 3 — Arquitetura (~2 min)

“A arquitetura é cliente-servidor em três blocos: Next.js no navegador, Spring Boot expondo REST na porta 8081, e PostgreSQL. O front não fala com o banco direto — só com a API. Isso separa experiência do usuário de regra de negócio e transação.”

“No Java, seguimos o encadeamento clássico: controller recebe JSON, service executa a operação com transação, repository persiste. Um exemplo forte é criar pedido: em um único fluxo transacional atualizamos estoque, gravamos movimentos e criamos a entrega. Se algo falhar, o banco desfaz — conceito de atomicidade.”

“No TypeScript, o arquivo api.ts concentra as chamadas HTTP tipadas — é o nosso contrato documentado do lado do cliente.”

Aba 4 — Fluxo de negócio (~1,5 min)

“O fluxo começa nos cadastros e chega ao pedido. O sistema valida se há estoque; ao persistir, dispara automaticamente as saídas e a fila de entregas. Por fim, o dashboard consolida indicadores a partir desses mesmos dados — ou seja, uma única fonte de verdade no PostgreSQL, lida pela API e exibida no React.”
